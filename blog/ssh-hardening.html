How I Hardened SSH and Dropped Attack Noise by 99% on Production Servers

A practical, config-first breakdown of the nine OpenSSH settings I apply to every Linux server I manage — and the real numbers behind why each one matters.

Running CI/CD infrastructure means your servers are always on, always reachable, and — without the right configuration — always being probed.

I pulled the auth logs on one of our build servers and found over 50,000 failed SSH attempts in a single month. Automated scanners cycling through credential lists, hammering port 22 around the clock. No targeted attack — just the background noise of the internet.

After applying these nine configurations consistently across our fleet, that number collapsed. Here's precisely what I changed, why it works, and what the real-world numbers looked like before and after.

1
Remove the Root Target Entirely
Automated scanners almost universally try root first. Make it a dead end.
🚨 Permitting root over SSH collapses your entire authentication model into a single point of failure. One cracked credential and the attacker has full, unrestricted access — no second layer, no audit trail, no recovery.
⚙️ /etc/ssh/sshd_config
PermitRootLogin no           # disable root login entirely

# If root access is needed only for automation (keys only, no passwords)
PermitRootLogin prohibit-password
The two-layer model
When you require a named user account with sudo, you introduce a mandatory second layer. An attacker must now compromise both the SSH account and the sudo password. Every sudo invocation is also recorded by PAM, giving you a proper audit trail for post-incident analysis — something root login can never provide.
⚠️ Before you apply this: open a second SSH session and confirm your non-root user can run sudo -i without issues. Only then reload sshd. I have learned this lesson the hard way on a cloud instance with no console access.
2
Replace Passwords with Cryptographic Keys
This single change made 99.7% of our observed brute-force attempts instantly pointless.
The fundamental problem with passwords
A password is a shared secret — it lives on both sides of the connection. That means it can be guessed, phished, leaked in a breach, or reused. An SSH key pair works on asymmetric cryptography: your private key never leaves your machine. An attacker can try for decades and get nothing without it.
⚙️ /etc/ssh/sshd_config
PubkeyAuthentication            yes
PasswordAuthentication          no
PermitEmptyPasswords            no
ChallengeResponseAuthentication no
UsePAM                          yes

Setup in 3 commands

💻 Terminal — Generate a modern ED25519 key
# 1. Generate a modern ED25519 key (fast, secure, modern standard)
ssh-keygen -t ed25519 -a 100 -C "your_email@example.com"

# 2. Copy the public key to your server
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server

# 3. Test in a SECOND terminal before closing the first
ssh -i ~/.ssh/id_ed25519 user@server
Use CaseKey Strategy
Work serversPassword-protected key (-a 100 rounds)
Personal serversSeparate key per server
CI/CD pipelinesRestricted key with command="" forced commands
-a 100 is the key derivation iteration count. Higher value = slower passphrase verification = much harder to brute-force offline if your private key file is ever stolen.
3
Stop Advertising Your SSH Port to the Internet
Port 22 is the first thing every scanner checks. Moving off it doesn't add security — it removes noise.

99.98%

⚙️ /etc/ssh/sshd_config
Port 2897    # pick any non-standard port above 1024
🔥 Firewall — Update UFW before restarting SSH
# Allow the NEW port first
sudo ufw allow 2897/tcp

# Remove the old rule
sudo ufw delete allow 22/tcp

sudo ufw enable
⚠️ Order matters: Always open the new port in your firewall before removing port 22, and always test from a second active terminal before rebooting sshd. A locked-out cloud VPS is expensive to fix.
Reducing noise is a genuine security benefit
A sophisticated, targeted attacker will port-scan and find your service regardless. But the vast majority of internet scanning is automated, dumb, and only checks port 22. Removing 99.98% of log noise means your monitoring tools surface real anomalies faster, and fail2ban isn't burning cycles on bot floods.
4
Make Every Failed Attempt Cost the Attacker Time
Five directives that stretch a brute-force campaign from hours into months.
⚙️ /etc/ssh/sshd_config
MaxAuthTries  3       # disconnect after 3 failed attempts per connection
MaxSessions   2       # max concurrent sessions per connection
LoginGraceTime 30     # seconds to authenticate before connection is dropped
MaxStartups   10:30:60 # start refusing at 10, drop 30% until full at 60
DirectiveWhat it doesAttack impact
MaxAuthTries 3 Drops connection after 3 wrong attempts Forces re-connection overhead per guess
LoginGraceTime 30 Kills unauthenticated connections after 30s Limits connection-holding DoS attacks
MaxStartups 10:30:60 Random early drops when connections pile up Defeats parallel brute-force floods
MaxStartups 10:30:60 means: start refusing with 30% probability once 10 unauthenticated connections exist, and refuse 100% once 60 are pending. This is your built-in DDoS protection.
5
Enforce SSH Access by Group, Not by Default
Default-deny at the user level. SSH access is a privilege, not a right.
⚙️ /etc/ssh/sshd_config
AllowGroups sshusers admins    # only these groups can SSH
💻 Terminal — Group setup
# Create SSH access group
sudo groupadd sshusers

# Grant a user SSH access
sudo usermod -aG sshusers alice

# Check group membership
groups alice
GroupPurposeExample users
sshusersNormal SSH accessDevelopers, ops team
adminsPrivileged access + 2FASenior eng, SRE
DenyUsersExplicit blockCompromised / departed accounts
Why service accounts should never have SSH access
Daemons like www-data, postgres, and nginx run with reduced privileges by design. If one is compromised via an application vulnerability, AllowGroups ensures that compromise stays contained — no interactive shell, no lateral movement via SSH, no escalation path.
6
Turn Off Every Feature You Haven't Explicitly Enabled
OpenSSH ships with a lot of features. Most CI/prod setups need almost none of them.
⚙️ /etc/ssh/sshd_config — Default lockdown
X11Forwarding          no   # GUI forwarding — almost never needed in CI/prod
AllowTcpForwarding     no   # blocks SSH tunnels / SOCKS proxying
AllowAgentForwarding   no   # blocks forwarding of SSH agent socket
PermitTunnel           no   # blocks tun/tap device forwarding
GatewayPorts           no   # prevents remote machines binding forwarded ports
IgnoreRhosts           yes  # ignores .rhosts/.shosts (legacy, insecure)
HostbasedAuthentication no  # no host-based auth trust
⚙️ /etc/ssh/sshd_config — Conditional exception for developers
# Surgical exceptions — only where genuinely needed
Match User developer
    AllowTcpForwarding  yes
    PermitOpen          localhost:5432 localhost:3306 # DB tunnels only
💡 The Match block is powerful. You can grant exceptions to specific users without globally opening attack surfaces. Default: everything locked. Exceptions: only where documented and approved.
Agent Forwarding is a lateral movement risk
When agent forwarding is enabled, a compromised jump server can silently use your SSH agent socket to authenticate onward to other hosts — without ever handling your private key directly. This makes it a clean lateral movement vector. Use ProxyJump in your client config (~/.ssh/config) instead — it achieves the same multi-hop without exposing the agent.
7
Explicitly Whitelist Your Cipher Suite
OpenSSH's defaults err on the side of compatibility. Production servers should err on the side of security.
⚙️ /etc/ssh/sshd_config — Modern crypto suite
Ciphers                chacha20-poly1305@openssh.com,aes256-gcm@openssh.com
MACs                   hmac-sha2-512-etm@openssh.com,hmac-sha2-256-etm@openssh.com
KexAlgorithms          curve25519-sha256
HostKeyAlgorithms      ssh-ed25519,rsa-sha2-512
PubkeyAcceptedAlgorithms ssh-ed25519,rsa-sha2-512
💻 Terminal — Regenerate host keys (remove old DSA/RSA)
# Remove legacy weak host keys
sudo rm /etc/ssh/ssh_host_dsa_key* 2>/dev/null
sudo rm /etc/ssh/ssh_host_rsa_key* 2>/dev/null
sudo rm /etc/ssh/ssh_host_ecdsa_key* 2>/dev/null

# Generate a clean ED25519 host key
sudo ssh-keygen -t ed25519 -f /etc/ssh/ssh_host_ed25519_key -N ""
Algorithm Type❌ Avoid✅ Use
CipherAES-CBC, RC4, 3DESChaCha20-Poly1305, AES-256-GCM
MACHMAC-MD5, HMAC-SHA1HMAC-SHA2-512-ETM, HMAC-SHA2-256-ETM
Key ExchangeDH group1, group14Curve25519-SHA256
Host KeyDSA, ECDSA, RSA-SHA1ED25519, RSA-SHA2-512
⚠️ Clients connecting for the first time after you regenerate host keys will get a "WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED" message. That's expected — they must accept the new fingerprint. This is not an attack.
8
Close the Feedback Loop: Logs Into Automated Bans
Verbose logging gives you forensic visibility. fail2ban turns that visibility into live firewall rules.
⚙️ /etc/ssh/sshd_config
LogLevel VERBOSE   # logs key fingerprints on every login
Where to find SSH logs
Ubuntu / Debian /var/log/auth.log
RHEL / CentOS / Amazon Linux /var/log/secure

With VERBOSE, successful logins include the key fingerprint used — enabling post-incident forensics if a key is ever compromised.
⚙️ /etc/fail2ban/jail.local — SSH jail config
[sshd]
enabled  = true
port     = 2849          # match your custom SSH port
maxretry = 3            # ban after 3 failures
bantime  = 3600         # ban for 1 hour (increase for repeat offenders)
findtime = 600          # look back 10 minutes
💻 Terminal — Install & check fail2ban
sudo apt install fail2ban

# Check current banned IPs
sudo fail2ban-client status sshd

# Unban an IP (e.g. if you locked yourself out)
sudo fail2ban-client set sshd unbanip 1.2.3.4
The combination is what makes this powerful. VERBOSE logging records enough context for forensic analysis. fail2ban watches those same logs in real time and inserts iptables DROP rules automatically. The loop is: OpenSSH logs the failure → fail2ban reads the log → kernel firewall silences the source IP.
9
Add a Second Factor for Privileged Accounts
A key pair can be stolen. A TOTP code expires in 30 seconds and requires physical device access.
When to use 2FA
Production servers Internet-facing hosts Bastion / jump servers Admin accounts

Service accounts and CI/CD pipelines should generally not use 2FA — they use restricted keys with forced commands instead.
⚙️ /etc/ssh/sshd_config — Require both key and OTP
AuthenticationMethods publickey,keyboard-interactive
💻 Terminal — Google Authenticator PAM setup
# Install the PAM module
sudo apt install libpam-google-authenticator

# Run setup interactively (per user)
google-authenticator

# Scan the QR code with your TOTP app (Google Authenticator, Authy, 1Password)
⚙️ /etc/ssh/sshd_config — Selective 2FA for admins only
# Regular users: key only (no 2FA friction)
AuthenticationMethods publickey

# Admin group: key + TOTP required
Match Group admins
    AuthenticationMethods publickey,keyboard-interactive
💡 This is the correct layering: reduce friction for regular developer workflows while ensuring that accounts with sudo privileges require demonstrable physical access to a registered device. Least privilege extended to the authentication mechanism itself.

All nine tricks combined into one drop-in configuration. Copy it, adjust your port and groups, then validate and reload.

Validation & Reload

💻 Terminal — Always validate before reloading
# 1. Test config syntax (catches typos before restart)
sudo sshd -t

# 2. Reload safely (no active sessions dropped)
sudo systemctl reload sshd

# 3. Verify connection in a SECOND terminal first
ssh -vvv user@server -p 2849

# 4. Check fail2ban is watching the right port
sudo fail2ban-client status sshd
Use reload, not restart. systemctl restart sshd drops every active SSH session immediately. reload applies the new config without touching existing connections — giving you a recovery window if something goes wrong.
The right way to think about server hardening
None of these configurations are individually airtight. What they do collectively is raise the cost and complexity of attacking your infrastructure to the point where automated scanners move on and targeted attackers must invest significantly more effort.

Security on a server isn't a finished state — it's an ongoing posture. Start with the highest-impact changes (keys and root login), build from there, and treat your audit logs as a first-class operational signal, not an afterthought.

Advanced Hardening

The nine core techniques get you 95% of the way there. These five go further — particularly useful for production fleets, bastion hosts, and multi-server environments.

A1
Kill Idle Sessions Automatically
An authenticated idle session is an open door. Set a hard timeout.
⚙️ /etc/ssh/sshd_config
ClientAliveInterval  300   # send a keepalive probe every 5 minutes
ClientAliveCountMax  2     # drop after 2 missed probes = 10 min idle → disconnect
Why idle sessions are dangerous
A developer SSHes into a server, walks away, and leaves their terminal open. If that laptop is then lost or stolen, the active session provides immediate access with no further authentication. ClientAliveInterval terminates the TCP connection at the server side after inactivity — regardless of what the client does.
A2
Bind SSH Keys to Source IP Addresses
A stolen key used from the wrong IP is silently rejected before auth completes.
📜 ~/.ssh/authorized_keys — per-key IP restriction
# Only allow this key from specific IPs or subnets
from="10.0.0.5,192.168.1.0/24" ssh-ed25519 AAAA... rishabh@workstation

# CI runner key — locked to runner IP only
from="10.0.2.10",no-agent-forwarding,no-pty,command="/usr/local/bin/deploy.sh" ssh-ed25519 AAAA... ci-runner
💡 The command="" option forces a specific command to run when the key is used — regardless of what the client requests. Combined with IP restriction, a CI runner key can only run your deploy script, from a known IP, and nothing else.
A3
Make the SSH Config Immutable
If an attacker reaches the filesystem, your hardened config shouldn't be editable — even by root.
💻 Terminal — Lock down the config file
# Correct ownership and permissions first
sudo chown root:root /etc/ssh/sshd_config
sudo chmod 600 /etc/ssh/sshd_config

# Make the file immutable — even root cannot modify it
sudo chattr +i /etc/ssh/sshd_config

# To edit it later, remove the immutable flag first
sudo chattr -i /etc/ssh/sshd_config
What chattr +i actually does
The immutable flag is set at the filesystem (ext4/xfs) level, below the permission model. Even a process running as root cannot write, rename, delete, or create hard links to an immutable file without first explicitly removing the flag. This is useful defence against malware or scripts that escalate to root and attempt to re-enable password auth.
A4
Use an SSH Certificate Authority (CA)
The right solution for multi-server fleets. No more managing authorized_keys on every host.
💻 Terminal — Setup SSH CA (one-time)
# 1. Generate the CA keypair once — store it securely (not on any server)
ssh-keygen -t ed25519 -f /etc/ssh/ca_key -C "internal-ca"

# 2. Tell sshd to trust certificates signed by this CA
#    Add to /etc/ssh/sshd_config on every server
TrustedUserCAKeys /etc/ssh/ca_key.pub

# 3. Sign a user's public key (valid for 8 hours)
ssh-keygen -s /etc/ssh/ca_key -I "rishabh" -n rishabh -V +8h ~/.ssh/id_ed25519.pub

# The signed cert is saved as ~/.ssh/id_ed25519-cert.pub
# SSH picks it up automatically — no authorized_keys change needed
ApproachProsCons
authorized_keys Simple, no infra needed Must update every server; hard to revoke at scale
SSH CA Instant revocation; certs expire automatically; no per-server key management Requires a CA key to be managed securely
Short-lived certificates (e.g. -V +8h) mean a stolen cert becomes worthless after 8 hours without any revocation action needed. This is how large infrastructure teams at scale handle SSH access — Cloudflare, Lyft, and Uber have all published their CA-based SSH workflows.
A5
Bind SSH to an Internal Interface Only
The strongest possible posture — SSH becomes physically unreachable from the public internet.
⚙️ /etc/ssh/sshd_config
# Only listen on the private/VPN interface — not the public IP
ListenAddress  10.0.1.25    # replace with your private interface IP
🔥 Firewall — Block SSH on public interface entirely
# Allow SSH only from VPN / internal subnet
sudo ufw allow from 10.0.0.0/8 to any port 2897

# Deny all other SSH traffic (overrides any default allow)
sudo ufw deny 2897/tcp
When to use this
Ideal for servers sitting inside a VPC (AWS, GCP, Azure) that are reachable via a VPN or bastion host. Users connect to the VPN first, then SSH over the private network. The public interface has no SSH listener at all — scanners cannot even establish a TCP connection, let alone attempt authentication.
💡 Priority order for these 5: ClientAliveInterval is a 2-line no-brainer — add it now. IP-restricted keys apply next time you rotate credentials. The SSH CA approach is worth investing in once you manage more than 5 servers.
