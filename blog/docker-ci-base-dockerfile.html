Breaking Down the docker-ci-base Dockerfile

A CI base image is the foundation every pipeline job runs on. Get it wrong and you're debugging obscure environment differences in production. Get it right and your pipelines are fast, reproducible, and self-contained. Let's tear apart the docker-ci-base Dockerfile layer by layer.

What is this image for?

This is a multi-purpose CI runner base image — a single Docker image that ships every tool a pipeline might need: Docker CLI, Java 21, Maven, Python 3, Ansible, JFrog CLI, and more.

The Base Image: docker:27-cli

The very first instruction sets the entire foundation of the image.

🐳 Dockerfile — line 1
FROM docker:27-cli
Why docker:27-cli and not ubuntu or debian?
docker:27-cli is an official Docker image that ships only the Docker CLI (no daemon), built on top of Alpine Linux. It is:

Tiny — Alpine base is ~5 MB vs ~70 MB for ubuntu:24.04
Docker-ready — the docker binary is already installed
Secure — minimal attack surface; fewer pre-installed packages
apk-based — Alpine's package manager is fast and cache-aware
Why pin to 27 and not latest?
Pinning to a major version (27) provides a balance: you get patch and minor updates automatically (security fixes) but you won't be surprised by a major version bump breaking docker build syntax or flags mid-sprint. Using latest in a base image is a reliability anti-pattern.

Alpine vs. Debian/Ubuntu comparison

PropertyAlpine (this image)Ubuntu/Debian
Base size ~5 MB ~70–120 MB
Package manager apk — fast, atomic apt — feature-rich, slower
C library musl libc glibc
Shell ash (default), bash installable bash pre-installed
Security Minimal surface, no systemd More services pre-installed
⚠ musl vs. glibc gotcha: Some pre-compiled binaries expect glibc and will segfault on Alpine. That's why musl-dev and linux-headers are installed — they allow compiling C extensions (e.g. Python native modules) against musl.

System Packages via apk

A single chained RUN installs (and verifies) all system-level dependencies.

🐳 Dockerfile — lines 7–38
# ---------- Versions (single source of truth) ----------
ENV MAVEN_VERSION=3.9.11 \
    JAVA_VERSION=21 \
    M2_HOME=/opt/maven \
    MAVEN_HOME=/opt/maven

ENV PATH="${PATH}:${M2_HOME}/bin"

# ---------- System packages ----------
RUN apk upgrade --no-cache && \
    apk add --no-cache \
        linux-headers \
        musl-dev \
        python3 \
        python3-dev \
        py3-pip \
        gcc \
        curl \
        bash \
        git \
        rpm \
        ansible \
        jq \
        yq \
        openjdk${JAVA_VERSION} \
        npm \
        wget \
        ca-certificates && \
    java -version

Package breakdown by category

linux-headers musl-dev gcc python3 python3-dev py3-pip curl wget git bash ca-certificates jq yq npm openjdk21 ansible rpm
Why is everything in one RUN?
Docker creates a new image layer for every RUN instruction. If you split package installs across multiple RUN commands, you get more layers, a larger image, and potentially stale cache entries (apt/apk cache files stuck in an intermediate layer). Chaining with && \ keeps it a single layer — and --no-cache means apk never writes its index to disk.
Why apk upgrade --no-cache first?
The base image (docker:27-cli) may have been built weeks ago. Running apk upgrade before installing anything ensures all existing packages get their latest security patches. --no-cache means do not store the downloaded index in the image (saves ~8 MB per build).
The ENV VERSION pattern (single source of truth)
JAVA_VERSION=21 and MAVEN_VERSION=3.9.11 are declared as ENV variables at the top. This means:

• openjdk${JAVA_VERSION} at install time resolves to openjdk21
• The Maven download URL uses ${MAVEN_VERSION}
• To upgrade, change one line — no grepping through the file
• They are also available at container runtime as environment variables
java -version at the end of the RUN instruction is a build-time smoke test. If Java didn't install properly, the build fails immediately with a clear error instead of producing a broken image silently.

Installing Apache Maven

Maven isn't in the Alpine package repository at the required version, so it's downloaded and installed manually from the official Apache archive.

🐳 Dockerfile — lines 40–46
# ---------- Maven ----------
RUN wget -q "https://archive.apache.org/dist/maven/maven-3/${MAVEN_VERSION}/binaries/apache-maven-${MAVEN_VERSION}-bin.tar.gz" \
        -O /tmp/maven.tar.gz && \
    tar -xzf /tmp/maven.tar.gz -C /tmp && \
    mv "/tmp/apache-maven-${MAVEN_VERSION}" "${M2_HOME}" && \
    rm /tmp/maven.tar.gz && \
    mvn --version

Step-by-step walkthrough

Download the binary tarball
wget -q "https://archive.apache.org/.../${MAVEN_VERSION}/…-bin.tar.gz" -O /tmp/maven.tar.gz
Uses the pinned MAVEN_VERSION env var. -q suppresses progress output in CI logs. Downloaded to /tmp — a throwaway location.
Extract the archive
tar -xzf /tmp/maven.tar.gz -C /tmp
Extracts the .tar.gz into /tmp. Results in a directory like /tmp/apache-maven-3.9.11/.
Move to the final install path
mv "/tmp/apache-maven-${MAVEN_VERSION}" "${M2_HOME}"
Moves to /opt/maven (the value of $M2_HOME). This is the conventional Maven home path followed by tooling like Eclipse, IntelliJ, and Maven wrappers.
Clean up
rm /tmp/maven.tar.gz
Removes the downloaded archive in the same layer. Critical: if you did rm in a separate RUN the archive would still occupy space in the previous layer.
Verify the installation
mvn --version
Build-time smoke test. The mvn binary is on $PATH because of ENV PATH="${PATH}:${M2_HOME}/bin" set earlier.
PATH trick: ENV PATH="${PATH}:${M2_HOME}/bin" is declared before this RUN block. Docker evaluates ENV at build time, so by the time mvn --version runs, the shell already knows where to find Maven.
Why not use apk add maven?
Alpine's maven package often lags behind the latest stable release. For a CI base image where build reproducibility matters, pinning to an exact version (here 3.9.11) and downloading from the official Apache archive is far more reliable than trusting the distro package to have the right version.

Python Packages (PEP-668 Safe)

Python 3 is installed via apk, but pip packages are installed with a special flag to avoid conflicts with the system package manager.

🐳 Dockerfile — lines 48–56
# ---------- Python packages (PEP-668 safe) ----------
RUN pip3 install --upgrade pip --break-system-packages && \
    pip3 install --break-system-packages \
        "virtualenv==20.26.6" \
        selinux \
        requests \
        "ruamel.yaml" \
        bs4 && \
    pip3 cache purge
What is PEP 668 and why does it matter?
PEP 668 (introduced in Python 3.11+) marks the system Python as "externally managed." If you try to pip install without a virtual environment on these systems, pip throws an error to prevent you from breaking system packages that apk also manages.

--break-system-packages tells pip: "I know what I'm doing — install globally anyway." In a Docker CI image this is the right call: there's no user to protect, and we want the packages globally available to all pipeline steps.

Installed Python packages

PackageVersionWhy it's here
virtualenv ==20.26.6 (pinned) Create isolated Python envs inside pipeline jobs; pinned for reproducibility
selinux latest Python bindings for SELinux; needed by Ansible on SELinux-enabled hosts
requests latest HTTP client; most automation and release scripts depend on it
ruamel.yaml latest YAML parser that preserves comments and order; used for K8s manifest manipulation
bs4 (BeautifulSoup4) latest HTML parsing; useful for scraping release notes or parsing HTML reports in CI
pip3 cache purge at the end removes the pip download cache from the image layer. pip caches downloaded wheels to speed up future installs, but in a baked image there won't be any future installs — so the cache is pure waste.
Why is only virtualenv pinned?
virtualenv has had breaking API changes in the past. Pinning ensures that pipeline scripts calling virtualenv programmatically get predictable behaviour. Libraries like requests and bs4 are much more stable across minor versions and benefit from receiving security patches automatically.

JFrog CLI Installation

The JFrog CLI (jf) is the command-line interface to Artifactory, Xray, Pipelines, and the broader JFrog Platform. It's installed via the official install script.

🐳 Dockerfile — lines 58–60
# ---------- JFrog CLI (latest stable) ----------
RUN curl -fL https://install-cli.jfrog.io | sh && \
    jf --version
What does curl -fL https://install-cli.jfrog.io | sh do?
-f — fail silently on HTTP errors (non-2xx status = exit code 22, not a silent empty script)
-L — follow redirects (the URL redirects to the latest release script)
| sh — pipes the downloaded shell script directly into sh for execution

The JFrog install script auto-detects the target architecture (linux-amd64, linux-arm64, etc.), downloads the correct binary, and places it in /usr/local/bin/jf.
⚠ Curl-to-shell: Piping a URL directly into sh is a common and convenient pattern, but it gives the script full control of the environment. For a CI base image this is generally acceptable (the script is from JFrog's own domain), but in security-sensitive environments you may want to download, verify the checksum, then execute.
Why "latest stable" instead of a pinned version?
The commented-out lines at the top of the file hint at a JFrog Artifactory integration. JFrog CLI has strong backwards compatibility guarantees, and using latest-stable means the image automatically picks up bug fixes and new Artifactory API support. If strict reproducibility is needed, the install URL can accept a specific version: https://releases.jfrog.io/artifactory/jfrog-cli/v2/[version]/jfrog-cli-linux-amd64/jf.
What can jf do in CI pipelines?
jf rt upload — push artifacts to Artifactory
jf rt download — pull dependencies from Artifactory
jf rt build-publish — publish build info (dependency graph, env vars, timestamps)
jf xr scan — trigger Xray security scans
jf c add — register Artifactory server credentials (used in entrypoint.sh)

Workspace, Config & Entrypoint

The final instructions set the working directory and configure the container entry point — currently partially commented out.

🐳 Dockerfile — lines 62–70
# ---------- Workspace ----------
WORKDIR /

# ---------- Config & entrypoint ----------
COPY settings.xml entrypoint.sh ./
RUN cp /settings.xml "${M2_HOME}/conf/settings.xml" && \
    chmod +x entrypoint.sh

ENTRYPOINT ["./entrypoint.sh"]
WORKDIR /
Sets the default working directory for subsequent RUN, COPY, CMD, and ENTRYPOINT instructions to the filesystem root /. This is unconventional (most images use /app or /workspace) but fine for a CI base image where pipeline jobs mount their workspace at runtime via volume mounts or GitLab's automatic clone path.

The entrypoint.sh script

📜 entrypoint.sh
#!/bin/bash

jf c add gamesys.jfrog.io \
  --interactive=false \
  --basic-auth-only \
  --url=https://abcd.jfrog.io/ \
  --user=$JFROG_USER \
  --password=$JFROG_TOKEN

exec "$@"
What jf c add does
This registers an Artifactory server with the JFrog CLI config store. After this runs, all subsequent jf commands (upload, download, build-publish) automatically know how to authenticate against gamesys.jfrog.io.

--interactive=false — run non-interactively (required in CI)
--basic-auth-only — use username+password (not API key or access token)
--user / --password — read from runtime environment variables
The exec "$@" pattern
exec "$@" replaces the shell process with whatever command was passed to docker run. This means:

• The entrypoint runs setup (JFrog auth), then hands off cleanly
• The main process gets PID 1 (critical for correct signal handling in containers)
• You can still override the command: docker run ci-base mvn clean package
The ENTRYPOINT and COPY commands enable the auto-auth behaviour, which is vital when the image becomes a properly configured CI base in a real GitLab/Jenkins environment.

Understanding settings.xml

📜 settings.xml — Maven Config & Credentials
<server>
  <id>server-1</id>
  <username>${env.JFROG_USER}</username>
  <password>${env.JFROG_TOKEN}</password>
</server>
<!-- ... -->
<repository>
  <id>server-1</id>
  <url>https://abcd.jfrog.io/artifactory/server-1</url>
</repository>
Servers vs. Repositories (The ID Mapping)
Our updated file maps generic endpoints (server-1, server-2, gitlab-server). The core concept relies on matching IDs: The <server> block acts as a vault containing authentication credentials injected dynamically by the CI runner (via JFROG_USER/CI_JOB_TOKEN). When Maven accesses a <repository> sharing the exact same <id>, it securely attaches those credentials.
Repositories vs. Plugin Repositories
Maven strictly separates Application Dependencies (libraries fetched by <repositories> to run your app) from Build Tools (compilers/execution plugins fetched by <pluginRepositories>). Even if both sets of files live natively on your single Artifactory abcd.jfrog.io instance, you must redeclare identical URLs into both XML blocks to grant Maven permission to access them for their respective pipeline purposes.

The Complete Dockerfile

Here's the entire file annotated inline for quick reference.

🐳 docker-ci-base/dockerfile — full file (70 lines)
# ── Base: Alpine-based Docker CLI image (v27) ──────────────────────────────
FROM docker:27-cli

# Credentials passed at RUNTIME only — never bake secrets into images
ARG JFROG_USER
ARG JFROG_TOKEN
ENV JFROG_USER=${JFROG_USER} \
    JFROG_TOKEN=${JFROG_TOKEN}

# ── Versions — single source of truth ─────────────────────────────────────
ENV MAVEN_VERSION=3.9.11 \
    JAVA_VERSION=21 \
    M2_HOME=/opt/maven \
    MAVEN_HOME=/opt/maven

ENV PATH="${PATH}:${M2_HOME}/bin"   # adds /opt/maven/bin to PATH

# ── System packages ────────────────────────────────────────────────────────
# One RUN = one layer. Upgrade first, then add. --no-cache keeps image lean.
RUN apk upgrade --no-cache && \
    apk add --no-cache \
        linux-headers musl-dev          # compile native C extensions against musl
        python3 python3-dev py3-pip      # Python 3 + pip
        gcc curl bash git wget           # essential build / transfer tools
        ca-certificates                 # TLS trust store for HTTPS requests
        rpm ansible jq yq               # RPM builds, infra automation, JSON/YAML
        openjdk${JAVA_VERSION}           # Java 21 (resolves env var at build time)
        npm &&                           # Node.js package manager
    java -version                        # smoke test ─ fail fast if Java broken

# ── Apache Maven 3.9.11 ────────────────────────────────────────────────────
# Download from official archive, extract, install to $M2_HOME, clean up.
RUN wget -q "https://archive.apache.org/dist/maven/maven-3/${MAVEN_VERSION}/binaries/apache-maven-${MAVEN_VERSION}-bin.tar.gz" \
        -O /tmp/maven.tar.gz && \
    tar -xzf /tmp/maven.tar.gz -C /tmp && \
    mv "/tmp/apache-maven-${MAVEN_VERSION}" "${M2_HOME}" && \
    rm /tmp/maven.tar.gz &&              # must rm in same RUN to save space
    mvn --version                        # smoke test

# ── Python packages (PEP-668 safe) ────────────────────────────────────────
RUN pip3 install --upgrade pip --break-system-packages && \
    pip3 install --break-system-packages \
        "virtualenv==20.26.6"            # pinned for API stability
        selinux requests                # Ansible SELinux + HTTP client
        "ruamel.yaml" bs4 &&             # YAML (comment-preserving) + HTML parser
    pip3 cache purge                     # remove cached wheels from image

# ── JFrog CLI (latest stable) ─────────────────────────────────────────────
RUN curl -fL https://install-cli.jfrog.io | sh && \
    jf --version                         # smoke test

# ── Workspace ─────────────────────────────────────────────────────────────
WORKDIR /

# ── Config & entrypoint ───────────────────────────────────────────────────
COPY settings.xml entrypoint.sh ./
RUN cp /settings.xml "${M2_HOME}/conf/settings.xml" && \
    chmod +x entrypoint.sh
ENTRYPOINT ["./entrypoint.sh"]
Every RUN block ends with a smoke-test command (java -version, mvn --version, jf --version). This is the fastest way to catch a broken install: the build fails loudly at the right step instead of producing a silently broken image.
